Documentação de Microserviços

Introdução

Microserviços são um estilo de arquitetura de software onde uma aplicação é construída como uma coleção de pequenos serviços independentes. Cada serviço se concentra em fazer uma coisa bem, executa em seu próprio processo e se comunica com outros serviços através de interfaces simples e bem definidas (geralmente APIs). Essa abordagem permite maior flexibilidade, escalabilidade e manutenção mais fácil em comparação com arquiteturas monolíticas tradicionais. Os microserviços permitem que as equipes desenvolvam, implantem e escalem diferentes partes de uma aplicação de forma independente, frequentemente usando diferentes tecnologias mais adequadas para cada serviço específico.

Pontos Principais:

  • Serviços Independentes: Cada serviço é responsável por uma função única, como gestão de usuários ou pagamentos.
  • Baixo Acoplamento: Os serviços podem ser desenvolvidos, implantados e escalados de forma independente.
  • Comunicação: Os serviços se comunicam entre si usando APIs (ex.: REST, gRPC).
  • Resiliência: Se um serviço falhar, os outros continuam funcionando.
  • Escalabilidade: Escale apenas os serviços que precisam, não a aplicação inteira.

Vantagens:

  • Escalabilidade: Serviços podem ser escalados individualmente.
  • Flexibilidade tecnológica: Diferentes serviços podem usar diferentes tecnologias.
  • Implantação independente: Cada serviço pode ser atualizado separadamente.
  • Resiliência: Falhas em um serviço não necessariamente afetam toda a aplicação.
  • Manutenção simplificada: Bases de código menores são mais fáceis de manter.

Desafios:

  • Complexidade distribuída: Gerenciar múltiplos serviços pode ser desafiador.
  • Consistência de dados: Manter dados consistentes entre serviços requer atenção.
  • Monitoramento: É necessário monitorar múltiplos serviços.
  • Testes: Testar interações entre serviços pode ser complexo.

Implementação:

  • Frequentemente utiliza-se containerização (ex: Docker) para implantação.
  • Orquestração de containers (ex: Kubernetes) para gerenciamento.

Contraste com arquitetura monolítica:

Em vez de uma única unidade, a aplicação é dividida em serviços menores e independentes.

Padrões comuns:

  • API Gateway
  • Service Discovery
  • Circuit Breaker
  • Event Sourcing
